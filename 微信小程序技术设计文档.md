# 微信小程序技术设计文档

## 重新思考与前端优先功能逻辑结构（贴合旅居场景）

说明：遵循“先思考后实现”，以用户习惯和使用场景为导向，先前端后后端，重构逻辑结构、环境与实现细节，并形成清晰的功能模块逻辑树与验收标准。

### 一、思考框架与使用场景
- 用户角色与习惯
  - 普通会员：微信授权即用，偏向热门活动/线路，倾向页面/线路/活动分享与快捷下单。
  - 领队/核销员：移动端扫码核销高频，关注核销效率、反核销与记录追踪。
  - 推广人员：二维码/链接快速绑定与统计，单级绑定避免复杂层级关系与冲突。
  - 企业会员：账号登录与资料完善，具对公信息与权限差异；不可与个人会员手机号冲突。
  - 管理员：维护基地/线路/活动，管理核销与提现规则，进行数据看板与名单导出。
- 关键场景
  - 浏览发现：首页 Tab（线路/活动/基地），空数据不扰民，信息卡简洁。
  - 下单预约：库存/容量校验，支付/报名闭环，状态一致，异常提示清晰。
  - 核销执行：扫码核销与反核销，二维码有效期与权限校验，日志可追溯。
  - 推广绑定：首次打开或扫码即绑定，时间优先，单级且防重复与循环。
  - 成长变现：领队申请、佣金累计、满足条件后提现，明确规则与提示。
  - 企业与运营：企业登录与资料维护，后台数据看板与导出。

### 二、环境与技术栈
- 前端
  - 微信开发者工具；原生小程序 + TypeScript。
  - UI：`vant-weapp`/`weui-miniprogram`；骨架屏、懒加载与空态隐藏。
  - 状态：全局 `store`（user/wallet/orders/config）；持久化到 `storage`。
  - 网络：统一 `request`，拦截鉴权/错误；幂等重试与降级。
  - 能力：`wx.login`/`getUserProfile`/`scanCode`/分享/订阅消息；二维码展示。
- 后端
  - Node.js 18+，NestJS/Express；MySQL 8 + TypeORM；Redis。
  - 安全与支付：JWT/Session；WeChat Pay V3；签名与回调校验。
  - 运维：日志与审计；配置中心；导出服务；限流与告警。

#### 开发环境与调试（本地 Mock 联调）
- Mock 服务：`node backend/mock-server.js 3001`；调试辅助：`GET /healthz`、`GET /__routes`。
- BASE_URL：`miniprogram/config.js` 设为 `http://127.0.0.1:3001`（开发者工具）；真机联调使用 `https://<your-domain>` 并在小程序后台配置服务器域名白名单。
- 开发者工具：本地设置勾选“启用不校验合法域名、TLS 版本以及 HTTPS 证书”，然后重启并重新编译。
- 请求封装：`miniprogram/utils/request.js` 将 `wx.request` Promise 化以支持 `async/await`，并已实现错误码映射与重试策略（见 [SPEC-B]）。
  - Mock 错误响应：`backend/mock-server.js` 的订单相关接口（详情/核销/反核销/退款）已按 [SPEC-B] 返回 `{ code, message }`，便于前端统一 Toast 文案与重试控制。
  - Mock 新增接口：`POST /activities/:id/signup`（报名容量冲突返回 56001、参数错误返回 42201）；`POST /orders`（库存容量冲突返回 56001、参数错误返回 42201；成功返回 `pending_pay`）。
- 懒加载：`miniprogram/app.json` 配置 `lazyCodeLoading: "requiredComponents"`，按需注入组件；必要时在页面 JSON 声明占位组件以实现“用时注入”。
- 已知提示：关闭校验后会出现“未校验合法域名/TLS/HTTPS 证书”的警告，属预期，不影响本地联调。
- 真实设备联调：需 `HTTPS` 证书、服务器域名白名单、TLS 1.2+，且在 Windows 上开放端口或使用内网穿透工具。

（本节对应 [PROG-M7] 前端 MVP 的本地联调说明；订单模块联调参考 [MOD-ORD] 与附录接口 [SPEC-A]）

### 三、前端功能模块逻辑树（先前端）
- 登录与身份
  - 微信授权注册/登录（个人）
  - 企业账号登录（账号/密码/验证码）
- 首页
  - Tab：旅游线路 / 热门活动 / 旅居基地
  - 列表 -> 详情 -> 预约/预订/报名 -> 下单/支付
  - 推荐分享（页面/线路/活动）
- 订单
  - 我参与的 / 其他推荐
  - 订单详情：状态（已支付/已使用/已过期/已退订） + 核销二维码
  - 反核销状态展示（如适用）
- 会员信息
  - 基本信息：ID/头像/名称/电话/积分/会员二维码
  - 领队能力：扫码核销入口（仅领队显示）
  - 企业会员：账号登录与改密
  - 推广绑定：扫码/链接首次绑定（单级，时间优先）
- 我的
  - 我的订单（同订单模块）
  - 申请提现：余额、成员名单、收入记录、提现记录
  - 申请领队：条件说明弹窗 -> 提交申请
  - 企业登录入口；账号信息（头像/昵称/电话/密码）；关于协议
- 活动
  - 活动列表与详情；报名窗口/时间/容量；满员阻止与提示
- 基地
  - 基地列表与详情；关联线路展示；富文本内容
- 关键流程（前端视角）
  - 登录：个人（wx.login）/ 企业（账号登录） -> 完成绑定与初始化
  - 下单/预约：库存/容量校验 -> 支付或报名 -> 订单生成
  - 核销：展示二维码 -> 领队扫码 -> 状态更新；支持反核销
  - 推广绑定：首次进入或扫码 -> 单级绑定 -> 防重复校验
  - 领队申请：条件校验 -> 确认弹窗 -> 提交 -> 后台审核
  - 提现：条件校验 -> 申请 -> 后台审批 -> 到账/拒绝

### 四、后端功能模块逻辑树（后端随后）
- 用户与组织：登录、部门、角色、用户，RBAC 与审计
- 基地管理：基地 CRUD；字段与富文本；线路多对多关联
- 线路管理：上下架；核销人员（多位）；关联基地；字段完整
- 活动管理：报名窗口/容量；名单导出；满员校验
- 会员管理：列表（订单/核销/拓展/提现）；领队申请；类型/等级/积分/佣金设置；提现规则
- 订单管理：筛选、详情、退订/退款；核销/反核销；状态一致与日志
- 数据看板：基地/会员/领队/推广/订单/核销/收入/支出指标与曲线；导出
- 系统设置：提现最低值、积分比例、二维码有效期等参数；权限策略
- 用户中心（后台）：Home 内容管理（内外链）；管理员个人资料与改密
- 核心服务：鉴权、支付、核销、推广绑定、佣金结算、提现审批、导出、告警与审计

### 五、数据模型与约束（摘要）
- Base、Line、Activity、Member（含类型/角色/企业扩展/积分/佣金/referrer）、Order（verify_status）、VerificationLog、CommissionRule、CommissionSettlement、WithdrawalRequest、ReferralLink、Settings
- 约束示例：
  - 单级推广绑定：用户 `referrer_id` 只允许一次设置；记录 `ref_bind_time`；禁止自引用与循环。
  - 佣金结算：线路过期且订单已核销才可生成结算；与提现解耦。
  - 核销幂等：同订单在有效期内核销请求具幂等键；反核销需权限与日志。
  - 报名容量：使用数据库行级锁防超卖（例如 `SELECT ... FOR UPDATE`）。

### 六、技术实现细节（摘要）
- 前端
  - 目录：`pages/*`、`components/*`、`store/*`、`utils/request.ts`、`services/*`
  - 请求封装：统一错误码/重试/节流；登录态续期
    - 已实现（[SPEC-B]）：`miniprogram/utils/request.js` 自动前缀 `BASE_URL`、指数退避重试（默认示例 `retry: 2`）、错误码映射到统一文案（401/403/404/429/5xx 及业务码 40101/40301/40401/40901/42201/42901/50000/54001/56001），统一 `wx.showToast` 提示；支持可选 `onUnauthorized` 回调处理未登录态。
  - 服务层（services）统一封装与切换策略
    - 已实现模块与方法：
      - `miniprogram/services/orders.js`：`list`、`getDetail`、`verify`、`unverify`、`refund`、`createOrder`
      - `miniprogram/services/activities.js`：`list`、`getDetail`、`signupActivity`
      - `miniprogram/services/lines.js`：`list`、`getDetail`
      - `miniprogram/services/bases.js`：`list`、`getDetail`
      - `miniprogram/services/api-registry.js`：集中管理接口路径（`orders|activities|lines|bases` 的 `list/detail/verify/unverify/refund/create/signup`）
    - 统一调用：全部通过 `miniprogram/utils/request.js`，默认 `retry: 2` 与 `toastOnError: true`，可选传入 `onUnauthorized`；接口查询参数统一使用 `toQS` 序列化，保持分页/筛选/排序一致性。
    - 接口映射（Mock → 页面）：
      - 订单：`GET /orders`（列表）、`GET /orders/:id`（详情）、`POST /orders/:id/verify|unverify|refund`、`POST /orders`（下单）
      - 活动：`GET /activities`（列表）、`GET /activities/:id`（详情）、`POST /activities/:id/signup`（报名）
      - 线路：`GET /lines`（列表）、`GET /lines/:id`（详情）
      - 基地：`GET /bases`（列表）、`GET /bases/:id`（详情）
    - 页面已切换（统一调用 services）：
      - `miniprogram/pages/orders/index.js` 使用 `ordersService.list`
      - `miniprogram/pages/orders/detail.js` 使用 `ordersService.getDetail|verify|unverify|refund`
      - `miniprogram/pages/lines/index.js` 使用 `linesService.list`
      - `miniprogram/pages/activities/index.js` 使用 `activitiesService.list`
      - `miniprogram/pages/bases/index.js` 使用 `basesService.list`
      - `miniprogram/pages/lines/detail.js` 使用 `linesService.getDetail`
      - `miniprogram/pages/activities/detail.js` 使用 `activitiesService.getDetail`
      - `miniprogram/pages/bases/detail.js` 使用 `basesService.getDetail`
    - 快速切换真实后端：
      - 更新 `miniprogram/config.js` 的 `BASE_URL` 指向真实域名；保留环境覆盖 `ENV_OVERRIDE`
      - 若路径或字段差异，仅在 `services/*` 或 `services/api-registry.js` 中调整 URL/参数/响应字段映射即可，页面保持不变
      - 如错误码或 `{ code, message }` 结构不同，可在 `request.js` 的错误映射或具体 `services` 方法中适配
      - 保持 `retry`、超时与未登录处理一致，避免页面分散逻辑
    - 错误行为：遵循 [SPEC-B]，`42201`（参数错误）、`56001`（容量/库存冲突）等业务码在服务层统一转换为友好提示，便于页面最小改动。
  - 组件：列表、卡片、二维码、扫码入口、弹窗、Tabs
  - 体验：骨架屏、懒加载、空态隐藏、超时提示
- 后端
  - 模块：`auth`、`members`、`lines`、`bases`、`activities`、`orders`、`verification`、`commissions`、`withdrawals`、`dashboard`、`settings`、`content`
  - 事务与并发：校验库存/容量与报名原子性；核销幂等；支付回调安全
  - 审计与导出：操作日志与名单/看板导出；权限检查
  - 配置中心：提现最低额、积分比例、二维码有效期等

### 七、验收标准（关键）
- 前端：空数据不扰民；状态与文案一致；扫码核销入口仅领队显示；单级绑定不重复；满员即时阻止报名；提现前置条件校验。
- 后端：权限与审计全覆盖；幂等与事务无超卖；核销/反核销日志完整；结算与提现规则严格执行；导出正确。

## 基于《微信小程序功能需求.md》的需求映射与定制（旅居项目）

说明：本节将《微信小程序功能需求.md》中的后台与前端模块全面映射到既有技术方案，明确“为什么/怎么做/接口/数据/验收”，并对数据模型与接口进行扩展以满足旅居场景的线路、基地、活动、会员、核销、佣金等要求。

### 后台系统模块映射与实现

- 用户与组织
  - 为什么：支撑多角色运营与权限隔离。
  - 怎么做：登录+部门+角色+用户管理，RBAC 权限。
  - 接口：POST /api/auth/login，CRUD /api/departments，/api/roles，/api/users。
  - 数据：Department、Role、User（含角色关联）。
  - 验收：不同角色访问受限，审计日志记录。

- 基地管理
  - 为什么：作为线路与内容的承载页，提升转化。
  - 怎么做：基地 CRUD；与线路多对多关联；内容富文本。
  - 字段：id、province_city、banner[]、name、intro_richtext、facilities、surroundings、videos[]。
  - 接口：CRUD /api/bases；关联 /api/lines/:id/bases。
  - 验收：前端有数据则展示，无数据不展示；线路页展示关联基地。

- 线路管理（含上下架、核销人员、关联基地）
  - 为什么：核心售卖单元，承载订单与佣金。
  - 怎么做：线路 CRUD；上下架；设置核销人员（支持多位领队/员工）；关联多个基地。
  - 字段：line_id、name、logo、content_richtext、period、sold_count、available_count、verifier_ids[]、base_ids[]。
  - 接口：CRUD /api/lines；PUT /api/lines/:id/status；PUT /api/lines/:id/verifiers；PUT /api/lines/:id/bases。
  - 验收：上下架状态影响前端可见；核销权限仅在设为核销人员后生效。

- 活动管理（含报名与名单导出）
  - 为什么：非线路型营销与社群运营。
  - 怎么做：活动 CRUD；报名窗口与容量；报名名单导出。
  - 字段：activity_id、name、logo、content、activity_time_range、signup_time_range、capacity、signup_count。
  - 会员信息扩展（用于名单）：member_id、name、phone、level、commission_balance、registered_at。
  - 接口：CRUD /api/activities；POST /api/activities/:id/signup；GET /api/activities/:id/export。
  - 规则：报名满员时禁止报名；企业内部报名与对外报名区分预留。
  - 验收：名单导出可用；满员提示与阻断生效。

- 会员管理（列表/领队申请/提现审批/会员设置）
  - 为什么：统一会员成长、核销权限与佣金体系。
  - 怎么做：
    - 列表：订单记录、核销记录、拓展下线、提现记录。
    - 领队申请：后台审核通过后打标签，授予核销权限。
    - 同意/拒绝：提现申请审批。
    - 会员设置：类型（个人/企业）、等级（默认普通）、积分比例、佣金设置（个人/企业按线路配置）、提现规则。
  - 关键规则：
    - 提现条件：订单对应的线路过期且已核销；最低提现金额配置化。
    - 角色标签可并存：推广人/领队/员工。
    - 企业会员需补充单位名称、付款账号、联系人、联系电话。
  - 接口：GET /api/members；POST /api/members/:id/leader-apply；PUT /api/members/:id/roles；PUT /api/members/:id/settings；审核提现 POST /api/withdrawals/:id/approve|reject。
  - 验收：角色/佣金/提现规则按配置执行；反核销闭环可追溯。

- 订单管理（列表/退订/核销状态）
  - 为什么：交易履约与核销闭环。
  - 怎么做：订单筛选与详情；退订/退款；核销状态管理（已核销/未核销）。
  - 接口：GET /api/orders；POST /api/orders/:id/refund；POST /api/orders/:id/verify；POST /api/orders/:id/unverify。
  - 验收：核销可追溯；退订影响资金与库存；状态一致。

- 数据看板
  - 为什么：运营数据与趋势判断。
  - 怎么做：指标聚合与趋势曲线。
  - 指标：基地、会员、领队、推广、订单、核销、收入、支出。
  - 接口：GET /api/dashboard/*（分模块聚合）。
  - 验收：时间维度筛选与导出正常。

- 系统设置
  - 为什么：系统参数与安全策略集中化。
  - 怎么做：配置中心（提现最低金额、积分比例、二维码有效期等）；权限策略。
  - 接口：GET/PUT /api/settings/*。
  - 验收：配置生效与审计记录。

- 用户中心（Home 管理/个人信息）
  - 为什么：前端展示源与管理员个人资料维护。
  - 怎么做：Home 内容列表增删改（支持内外链）；个人信息与改密。
  - 接口：CRUD /api/home；PUT /api/users/:id/profile。
  - 验收：内容发布即前端可见；权限与审核链路可选。

### 小程序前端模块映射与实现

- 用户登录
  - 为什么：识别会员与企业账号。
  - 怎么做：个人走微信授权注册登录；企业使用账号+密码+验证码。
  - 接口：POST /api/auth/wechat-login，POST /api/auth/enterprise-login。
  - 规则：企业预留电话不可再注册为个人会员；个人需绑定手机号。
  - 验收：两类登录与账号状态区分明确。

- 首页（菜单栏/推荐分享）
  - 为什么：统一入口与转化。
  - 怎么做：Tabs：旅游线路/热门活动/旅居基地；对应列表与详情；全局分享。
  - 接口：GET /api/lines，GET /api/activities，GET /api/bases。
  - 验收：无数据隐藏；分享卡片与落地页正确。

- 订单（我参与的/其他推荐）
  - 为什么：订单管理与二次转化。
  - 怎么做：两个选项卡；详情含订单状态（已支付/已使用/已过期/已退订）、线路信息、核销二维码；他荐展示除已下单外其他线路。

说明：结合前文“权限矩阵”，以装饰器 + Guard 方式实现基于资源/动作的 RBAC，并支持“所有者约束”和业务条件校验（如反核销仅管理员/运营允许）。以下为 TypeScript 风格伪代码，便于快速落地。

```typescript
// 权限装饰器：标注资源与动作
export const Permission = (resource: string, action: string, options?: { owner?: boolean }) =>
  SetMetadata('permission', { resource, action, owner: !!options?.owner });

// 用户实体与角色（示意）
interface UserCtx { id: number; roles: string[]; type: 'personal'|'enterprise'; }

// 资源策略（示例）：与“附录C 权限矩阵”一致
const ResourcePolicy = {
  lines: {
    create: ['admin','operator'], update: ['admin','operator'], status: ['admin','operator'],
  },
  bases: { create: ['admin','operator'], update: ['admin','operator'] },
  activities: { create: ['admin','operator'], export: ['admin','operator'], signup: ['member','enterprise'] },
  orders: {
    listMine: ['member','enterprise','leader'],
    refund: ['member','enterprise'],              // 所有者约束：仅本人订单可退
    verify: ['leader'],                            // 领队核销
    unverify: ['admin','operator'],                // 反核销需管理员/运营
  },
  members: { list: ['admin','operator'], me: ['member','enterprise','leader','promoter'], bindReferrer: ['member','enterprise'] },
  commissions: { rules: ['admin','operator'], settle: ['admin','operator'], me: ['member','enterprise'] },
  withdrawals: { create: ['member','enterprise'], approve: ['admin','operator'] },
  dashboard: { overview: ['admin','operator'] },
  settings: { general: ['admin'] },
} as const;

// Guard：权限 + 所有者约束 + 业务条件
@Injectable()
export class PermissionGuard implements CanActivate {
  constructor(private reflector: Reflector, private orders: OrdersService) {}

  async canActivate(ctx: ExecutionContext): Promise<boolean> {
    const req = ctx.switchToHttp().getRequest();
    const user: UserCtx | undefined = req.user;            // 由 AuthGuard/JWT 预先注入
    if (!user) throw new UnauthorizedException('40101');

    // 读取装饰器元数据
    const meta = this.reflector.get<{resource:string, action:string, owner:boolean}>('permission', ctx.getHandler());
    if (!meta) return true; // 未标注则放行（也可默认拒绝）

    const allowed = ResourcePolicy[meta.resource]?.[meta.action] || [];
    if (!allowed.length) throw new ForbiddenException('40301');

    // 角色匹配
    const hasRole = user.roles.some(r => allowed.includes(r));
    if (!hasRole) throw new ForbiddenException('40301');

    // 所有者约束（例如：订单退款仅本人）
    if (meta.owner && meta.resource === 'orders' && meta.action === 'refund') {
      const orderId = Number(req.params.id);
      const order = await this.orders.findById(orderId);
      if (!order) throw new NotFoundException('40401');
      if (order.member_id !== user.id) throw new ForbiddenException('40301');
    }

    // 业务条件示例：反核销只能管理员/运营（已在角色层面拦截）；核销二维码有效期校验
    if (meta.resource === 'orders' && meta.action === 'verify') {
      const orderId = Number(req.params.id);
      const ok = await this.orders.checkQrCodeValid(orderId);
      if (!ok) throw new UnprocessableEntityException('42201');
    }

    return true;
  }
}

// 用法示例（Controller）
@Controller('orders')
@UseGuards(JwtAuthGuard, PermissionGuard)
export class OrdersController {
  @Post(':id/refund')
  @Permission('orders', 'refund', { owner: true })
  refund() { /* ... */ }

  @Post(':id/verify')
  @Permission('orders', 'verify')
  verify() { /* ... */ }

  @Post(':id/unverify')
  @Permission('orders', 'unverify')
  unverify() { /* ... */ }
}
```

思考要点：
- 权限来源统一：用资源/动作模型代替零散角色判断，便于管理矩阵与审计。
- 所有者约束：通过 `options.owner` 配合服务层数据查询实现“本人资源”限制。
- 业务条件：尽量在服务层实现（库存、容量、二维码有效期、审批流），Guard 只做轻量校验。

## 图片需求总结与模块释义（含“为什么”和“怎么做”）

### 用户侧模块
- 首页
  - 为什么：首屏承载发现与转化入口，减少跳转与困惑；活动与公告提高时效与参与。
  - 怎么做：
    - 在 `pages/index` 布局 Banner/公告/推荐宫格/分类入口/搜索框。
    - 并行拉取 `GET /api/categories` 与首页推荐 `GET /api/products?recommend=true`，展示骨架屏。
    - 弹窗位按配置控制（活动/新手引导），支持关闭与下次不再显示。
    - 点击卡片跳转 `pages/detail` 并预加载详情数据。

- 搜索
  - 为什么：满足目标性查找，配合热词提升效率。
  - 怎么做：
    - 在 `pages/index` 或独立 `pages/search` 提供输入框，记录历史搜索到 `storage`；支持清除。
    - 请求 `GET /api/products?keyword=xxx&page=&size=`，与类目筛选联动。
    - 弱网与空态处理（无结果提示、推荐替代）。

- 分类/列表
  - 为什么：结构化浏览与筛选，满足比较与选择。
  - 怎么做：
    - 在 `pages/catalog` 渲染类目 Tab 与筛选（价格区间/销量/评分）。
    - 分页加载，上拉触底继续；使用骨架屏与占位图。
    - 请求 `GET /api/products?category_id=&page=&size=&sort=`；保留滚动位置记忆。

- 详情页
  - 为什么：促成决策；规格与库存是下单前置校验。
  - 怎么做：
    - 在 `pages/detail` 展示图文、评价、规格选择（SKU）、价格、库存。
    - 规格弹层选择后更新当前 SKU；前端根据返回的 `sku.stock` 提示可售，最终以服务端订单创建校验为准。
    - 按场景提供「加入购物车」或「立即预约/下单」。

- 购物车（电商场景）
  - 为什么：提升客单与组合购买，支撑优惠聚合。
  - 怎么做：
    - `pages/cart` 管理条目（增删改/全选），登录前本地存储、登录后与服务端同步（`GET/POST /api/cart`）。
    - 校验限购与库存；结算进入 `pages/order-confirm`。

- 预约填写（服务场景）
  - 为什么：保障服务承载能力与到店/到时可交付。
  - 怎么做：
    - `pages/reserve` 选择服务、日期与时间段，备注与数量。
    - 拉取 `GET /api/reservations/slots?service_id=&date=` 获取容量与禁用时段。

- 确认订单
  - 为什么：支付前最后校验，减少错误与售后。
  - 怎么做：
    - `pages/order-confirm` 汇总地址/联系人/优惠券/发票/备注与金额明细。
    - 调用 `POST /api/orders` 创建订单，服务端进行价格/库存/优惠原子校验并返回应付金额与订单号。

- 支付
  - 为什么：完成交易闭环；回调更新确保一致性。
  - 怎么做：
    - 调用 `POST /api/pay/prepay` 以 `order_id` 获取 `timeStamp/nonceStr/package/signType/paySign`。
    - 调用 `wx.requestPayment`；成功后跳转 `pages/order-detail` 并提示订阅消息。
    - 失败提供重试与返回确认页；统一异常引导。

- 订单列表
  - 为什么：用户自助管理订单与下一步行动。
  - 怎么做：
    - `pages/order-list` 按状态筛选（全部/待支付/待服务/已完成/售后）。
    - 请求 `GET /api/orders?status=&page=&size=`；点击进入详情。

  - 怎么做：
    - `pages/order-detail` 展示状态流水、物流/排班、支付信息；支持取消/退款/再次购买。
    - 处理支付成功但回调延迟的情形：轮询或下拉刷新以服务端状态为准。

  - 为什么：聚合用户资产与偏好设置，减少二次输入。
  - 怎么做：

  - 怎么做：

### 关键流程分步说明（每一步的“为什么/怎么做”）
- 登录与鉴权
  - 为什么：降低首次摩擦，必要节点才授权；统一 `openid` 身份。
  - 怎么做：在提交订单或查看订单时触发 `wx.login`，后端 `POST /api/auth/login`（`code2session`）生成 `token`；手机号用 `getPhoneNumber` 二次绑定。

- 浏览与发现
  - 为什么：提高探索效率与首屏转化。
  - 怎么做：首页并行拉取类目与推荐；列表分页与筛选；详情预加载与骨架屏。

- 下单与支付
  - 为什么：保障价格与库存一致性，支付参数安全。
  - 怎么做：`POST /api/orders` 原子校验创建订单 → `POST /api/pay/prepay` → `wx.requestPayment` → 后端 `POST /api/pay/notify` 幂等更新状态。

- 预约与排班
  - 为什么：服务容量与人员安排需实时可控。
  - 怎么做：拉取可用时间段与容量，预约创建成功后进入订单/预约详情，并在接近服务时间推送提醒。

- 订阅消息与通知
  - 为什么：降低漏通知与提升到店/按时完成率。
  - 怎么做：关键节点请求订阅；服务端根据状态触发模板消息，记录推送结果与失败重试。

- 售后与退款
  - 为什么：保障用户权益与树立信任。
  - 怎么做：订单详情发起申请，后端审核与退款（原路退回），记录审计与资金流水。

- 数据与对账
  - 为什么：业务健康与财务合规。
  - 怎么做：支付与订单数据定时对账，异常告警；库存与预约容量一致性校验。


## 思考与假设
- 目标：用户在小程序内完成浏览/搜索/下单或预约/支付/订单管理/售后。
- 习惯：打开即用，尽量延迟授权（登录/手机号/订阅消息），支付流程最短。
- 架构：前端（微信小程序）+ 后端（Node.js + MySQL + 微信支付V3），可替换为微信云开发实现同等功能。

## 环境与技术栈
- 前端：微信开发者工具（基础库≥2.30），原生小程序（WXML/WXSS/JS/TS），UI库 `vant-weapp` 或 `weui-miniprogram`。
- 前端状态与网络：`App.globalData` + 轻量状态库（可用 mobx-miniprogram），统一封装 `wx.request`，处理 token 与错误。
- 后端：Node.js 18+（NestJS/Express）、MySQL 8、TypeORM，微信支付 V3；对象存储（COS）用于图片。
- 部署：HTTPS 域名与证书，API 域名在小程序后台配置；Docker 部署可选。

## 使用场景与用户习惯
- 首页直达核心内容；不强制登录，进入订单/支付时再触发授权。
- 支持分类/搜索/分页/下拉刷新，上拉加载；失败回退与重试明确。
- 支付后返回订单详情，提供订阅消息提醒。

## 前端设计
### 架构与页面结构
- 目录：
  - `app.ts/app.json` 全局配置与生命周期。
  - `pages/index` 首页（Banner/推荐/分类/搜索入口）。
  - `pages/catalog` 列表页（筛选/排序/分页）。
  - `pages/detail` 详情页（图文/规格/库存/评价）。
  - `pages/order-confirm` 确认订单（地址/优惠/备注/发票）。
  - `pages/order-list`/`pages/order-detail` 订单列表与详情。
  - `pages/profile` 我的（地址/优惠券/设置/客服/反馈）。
  - `pages/reserve`（预约场景）或 `pages/cart`（购物车场景）。
  - `components/*` 规格选择/卡片/骨架屏/空态等。

### 前端功能逻辑树（文本）
- 首页 → Banner/公告/推荐宫格/分类入口/搜索框
- 列表页 → 分类筛选/排序/分页加载/空态
- 详情页 → 基础信息/图文/评价/规格选择/库存校验/下单按钮
- 购物车/预约填写 → 编辑项/数量/时间/备注
- 确认订单 → 地址/联系人/优惠/发票/金额明细/提交
- 支付 → 拉起支付/结果处理/失败重试
- 订单 → 列表（全部/待付/待服务/已完成/售后）/详情（取消/退款/再次购买）
- 我的 → 个人信息/授权/消息订阅/地址/优惠券/客服/反馈
- 通知与分享 → 订阅消息/分享卡片/海报

### 关键流程
- 登录与鉴权：在需要时触发 `wx.login` → 后端 `code2session`（openid/session_key）→ 返回 `token` 存储于本地。
- 下单与支付：选择规格/校验库存与价格 → 创建订单（待支付）→ 预下单取支付参数 → `wx.requestPayment` → 成功更新订单状态 → 引导订阅消息。
- 预约流程：选择服务与时间段 → 创建预约订单 → （可选）预付 → 订阅提醒。

### 技术细节
- 网络封装：统一 `baseURL`/`token` 注入，响应码与错误提示，鉴权失效重登。
- 状态与缓存：登录态与用户信息 `globalData` + `storage`；列表/详情短时缓存，订单实时拉取。
- 授权与订阅：手机号 `getPhoneNumber` 按需；`wx.requestSubscribeMessage` 在下单/状态变更节点引导。

## 后端设计
### 架构与模块
- 分层：Controller（HTTP 接入）/Service（领域）/Repository（数据）/Infra（缓存/对象存储/队列/日志）。
- 核心模块：用户/商品与类目/SKU与库存/订单/预约/支付/通知/营销/系统配置。

### 后端功能逻辑树（文本）
- 用户：登录（openid）/资料/地址/手机号绑定
- 商品服务：类目/标签/SKU/规格/价格/库存/媒体
- 订单：创建/确认/支付/取消/完成/售后/退款
- 预约：时间段/排班/变更通知
- 支付：统一下单/回调/对账/退款
- 通知：订阅消息/站内信/短信
- 营销：优惠券/活动规则/发放/核销
- 系统：权限/日志/审计/配置/字典/任务

### 数据模型（关键字段示例）
- 用户 `user`：`id/openid/unionid/nickname/avatar/mobile/created_at`
- 地址 `address`：`id/user_id/consignee/mobile/location/is_default`
- 类目 `category`：`id/name/parent_id/sort`
- 商品 `product`：`id/title/category_id/status/cover_url/price_min/price_max`
- SKU `sku`：`id/product_id/spec_json/price/stock`
- 订单 `order`：`id/user_id/status/amount_total/amount_payable/coupon_id/prepay_id/pay_order_no`
- 订单项 `order_item`：`id/order_id/sku_id/title/spec_json/price/quantity`
- 预约 `reservation`：`id/user_id/service_id/date/time_slot/status`
- 支付流水 `payment`：`id/order_id/amount/channel/trade_state/transaction_id`

### 接口设计（简要）
- 鉴权：`POST /api/auth/login`（入参 `code`，出参 `token/user`）
- 用户：`GET/PUT /api/user/profile`；`GET/POST /api/user/address`
- 类目与商品：`GET /api/categories`；`GET /api/products`；`GET /api/products/:id`
- 订单：`POST /api/orders`；`GET /api/orders`；`GET /api/orders/:id`；`POST /api/orders/:id/cancel`
- 预约：`GET /api/reservations/slots`；`POST /api/reservations`；`GET /api/reservations`
- 支付：`POST /api/pay/prepay`；`POST /api/pay/notify`
- 通知：`POST /api/notify/subscribe`；`POST /api/notify/send`

### 支付与通知要点
- 微信支付 V3：统一下单（商户私钥签名，保存 `prepay_id`）；前端 `wx.requestPayment`；回调校验签名，幂等更新订单。
- 订阅消息：前端引导订阅，服务端按模板与 `openid` 推送。

## 运维与安全
- 安全：仅信任服务端支付参数；价格与库存校验在服务端；JWT 鉴权；订单创建去重键；限流与风控。
- 合规：隐私数据加密存储与审计；支付与状态变更全链路日志与对账。
- 部署与监控：证书与密钥安全（KMS/env）；接口失败率、支付回调异常、库存不一致告警。
